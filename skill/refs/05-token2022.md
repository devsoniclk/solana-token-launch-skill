# Token-2022 Extensions Guide

## Token-2022 vs Legacy SPL Token

| Feature              | SPL Token (legacy) | Token-2022          |
|----------------------|--------------------|---------------------|
| Program ID           | TokenkegQ...       | TokenzQd...         |
| Transfer fees        | ✗                  | ✓                   |
| Confidential transfers| ✗                 | ✓                   |
| Transfer hooks       | ✗                  | ✓                   |
| On-chain metadata    | ✗ (needs Metaplex) | ✓ (built-in)        |
| Non-transferable     | ✗                  | ✓                   |
| Interest-bearing     | ✗                  | ✓                   |
| Required memo        | ✗                  | ✓                   |
| Default account state| ✗                  | ✓                   |
| Permanent delegate   | ✗                  | ✓                   |
| CPI Guard            | ✗                  | ✓                   |
| Mint close authority | ✗                  | ✓                   |
| Composability        | Excellent          | Good (improving)     |
| Ecosystem support    | Universal          | Growing (2026)       |
| Compute cost         | Lower              | Slightly higher      |

### When to Use Token-2022

**Use Token-2022 if you need ANY of:**
- Transfer fees
- Confidential transfers
- Custom transfer logic (hooks)
- On-chain metadata without Metaplex
- Non-transferable (soulbound) tokens
- Required memos for compliance

**Use legacy SPL Token if:**
- Maximum ecosystem compatibility is priority
- No extensions needed
- Interacting with protocols that don't yet support Token-2022

**In 2026:** Token-2022 is production-ready and widely supported. Jupiter, Raydium, Orca all support it. Use it for new projects unless you have a specific reason not to.

## Transfer Fees

Automatically collect a fee on every token transfer. Fee is taken from the transfer amount.

### Setup

```bash
# Create mint with transfer fee extension
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --decimals 6 \
  --transfer-fee 50 5000   # 50 basis points (0.5%), max 5000 per tx

# The fee extension stores fees in a separate account
# Configure fee destination
spl-token authorize <MINT> transfer-fee-config <FEE_AUTHORITY>
```

### TypeScript

```typescript
import {
  createInitializeTransferFeeConfigInstruction,
  createMint,
  getMintLen,
  ExtensionType,
  TOKEN_2022_PROGRAM_ID,
} from "@solana/spl-token";

const extensions = [ExtensionType.TransferFeeConfig];
const mintLen = getMintLen(extensions);
const mintLamports = await connection.getMinimumBalanceForRentExemption(mintLen);

// Create mint account, initialize transfer fee, then initialize mint
const transferFeeConfigIx = createInitializeTransferFeeConfigInstruction(
  mint,
  feeAuthority,    // Can update fee config
  feeAuthority,    // Withdraw authority for collected fees
  transferFeeBasisPoints, // 50 = 0.5%
  maxFee,          // Max fee per transfer (in tokens)
  TOKEN_2022_PROGRAM_ID
);
```

### Fee Mechanics

```
Transfer amount: 1,000,000 tokens
Fee rate: 50 basis points (0.5%)
Fee collected: 5,000 tokens
Recipient receives: 995,000 tokens

Fees are withheld in the recipient's token account
Fee authority withdraws accumulated fees periodically
```

### When to Use

- Protocol revenue generation
- Auto-funding treasury on every transfer
- Transaction tax (meme tokens — use cautiously, can hurt adoption)

### Gotchas

- Fees reduce the amount the recipient gets — must be communicated
- Some DEXes may not route through tokens with transfer fees
- Fee withdrawal is a separate operation (batch periodically)
- Max fee cap prevents excessive fees on large transfers
- Cannot be removed once set on a mint

## Confidential Transfers

Hide transfer amounts while keeping the token balances encrypted. Uses zero-knowledge proofs.

### Setup

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --confidential-transfers auto
```

### How It Works

1. Token accounts have encrypted balance (elGamal encryption)
2. Each transfer includes a ZK proof that the transfer is valid
3. Balances are hidden from public view
4. Account owner can decrypt their own balance
5. Auditor key (optional) can decrypt all transfers for compliance

### TypeScript

```typescript
import {
  createInitializeConfidentialTransferMintInstruction,
  ExtensionType,
  getMintLen,
} from "@solana/spl-token";

const extensions = [ExtensionType.ConfidentialTransferMint];
const mintLen = getMintLen(extensions);

const initConfidential = createInitializeConfidentialTransferMintInstruction(
  mint,
  authority,       // Can configure auditor
  autoApprove,     // true = accounts auto-approved for confidential transfers
  TOKEN_2022_PROGRAM_ID
);
```

### When to Use

- Privacy-preserving payments
- Institutional use (hide treasury movements)
- Compliance with privacy regulations

### Gotchas

- Significantly higher compute costs per transfer (~100K+ CU)
- Not all DEXes fully support confidential transfers
- ZK proof generation adds latency
- Auditor key management is critical for compliance
- Can be combined with regular (non-confidential) transfers

## Default Account State

Force all new token accounts to be frozen by default. Users must be unfrozen before they can receive tokens.

### Setup

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --default-account-state frozen
```

### How It Works

1. All newly created token accounts start as `frozen`
2. The freeze authority must approve (unfreeze) each account
3. This enables allowlist/KYC-gated token distributions

### TypeScript

```typescript
import {
  createInitializeDefaultAccountStateInstruction,
  AccountState,
  ExtensionType,
  getMintLen,
} from "@solana/spl-token";

const extensions = [ExtensionType.DefaultAccountState];
const mintLen = getMintLen(extensions);

const initDefaultState = createInitializeDefaultAccountStateInstruction(
  mint,
  AccountState.Frozen, // or AccountState.Initialized
  TOKEN_2022_PROGRAM_ID
);
```

### When to Use

- Regulated securities tokens (KYC-gated)
- Compliance tokens where only approved wallets can transact
- Game items that need to be "activated" before use

### Gotchas

- Requires retaining freeze authority (centralization)
- Users can't receive tokens until unfrozen — bad UX if not automated
- Can be changed later by the authority (frozen ↔ initialized)

## Permanent Delegate

Grants a permanent, irrevocable delegate authority over all token accounts for a mint.

### Setup

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --permanent-delegate <DELEGATE_ADDRESS>
```

### TypeScript

```typescript
import {
  createInitializePermanentDelegateInstruction,
  ExtensionType,
  getMintLen,
} from "@solana/spl-token";

const extensions = [ExtensionType.PermanentDelegate];
const mintLen = getMintLen(extensions);

const initDelegate = createInitializePermanentDelegateInstruction(
  mint,
  delegateAddress,
  TOKEN_2022_PROGRAM_ID
);
```

### What It Does

The permanent delegate can:
- **Transfer** tokens from any account (without owner approval)
- **Burn** tokens from any account

This is extremely powerful and dangerous.

### When to Use

- Recoverable tokens (lost key recovery)
- Compliance tokens (freeze + seize for legal orders)
- Game tokens (developer needs to move items between accounts)
- Subscription tokens (revoke access on non-payment)

### Gotchas

- **Massive centralization risk** — the delegate can steal all tokens
- Cannot be removed once set
- Must be clearly communicated to users
- Consider combining with multisig for the delegate authority

## Transfer Hook

Execute custom program logic on every transfer. The most powerful extension.

### Setup

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --transfer-hook <HOOK_PROGRAM_ID>
```

### Hook Program (Anchor)

```rust
use anchor_lang::prelude::*;
use spl_tlv_account_resolution::account::ExtraAccountMeta;
use spl_transfer_hook_interface::instruction::{ExecuteInstruction, TransferHookInstruction};

declare_id!("HookProgramId111111111111111111111111111111");

#[program]
pub mod transfer_hook_example {
    use super::*;

    pub fn initialize_extra_account_metas(
        ctx: Context<InitializeExtraAccountMetas>,
    ) -> Result<()> {
        // Define extra accounts needed by the hook
        let account_metas = vec![
            ExtraAccountMeta::new_with_pubkey(
                &ctx.accounts.config.key(),
                false, // is_signer
                true,  // is_writable
            )?,
        ];

        spl_transfer_hook_interface::initialize_extra_account_metas(
            ctx.program_id,
            &ctx.accounts.extra_account_metas.to_account_info(),
            &ctx.accounts.mint.to_account_info(),
            &ctx.accounts.authority.to_account_info(),
            &ctx.accounts.system_program.to_account_info(),
            &account_metas,
        )?;

        Ok(())
    }

    pub fn transfer_hook(ctx: Context<TransferHook>, amount: u64) -> Result<()> {
        let config = &ctx.accounts.config;
        
        // Enforce max wallet
        let dest_account = &ctx.accounts.destination_token;
        require!(
            dest_account.amount + amount <= config.max_wallet_amount,
            HookError::ExceedsMaxWallet
        );

        // Enforce max transaction
        require!(
            amount <= config.max_tx_amount,
            HookError::ExceedsMaxTx
        );

        // Enforce cooldown
        let clock = Clock::get()?;
        let last_transfer = ctx.accounts.destination_token_data.last_transfer;
        require!(
            clock.unix_timestamp - last_transfer >= config.cooldown_seconds,
            HookError::CooldownNotMet
        );

        // Update last transfer time
        ctx.accounts.destination_token_data.last_transfer = clock.unix_timestamp;

        Ok(())
    }
}

#[derive(Accounts)]
pub struct TransferHook<'info> {
    #[account(
        token::mint = mint,
        token::authority = owner,
    )]
    pub source_token: Account<'info, TokenAccount>,
    
    pub mint: Account<'info, Mint>,
    
    #[account(
        token::mint = mint,
    )]
    pub destination_token: Account<'info, TokenAccount>,
    
    /// CHECK: Owner of source token
    pub owner: UncheckedAccount<'info>,
    
    #[account(mut)]
    pub config: Account<'info, HookConfig>,
    
    #[account(mut)]
    pub destination_token_data: Account<'info, TokenData>,
}

#[account]
pub struct HookConfig {
    pub max_wallet_amount: u64,
    pub max_tx_amount: u64,
    pub cooldown_seconds: i64,
}

#[account]
pub struct TokenData {
    pub last_transfer: i64,
}

#[error_code]
pub enum HookError {
    #[msg("Exceeds maximum wallet balance")]
    ExceedsMaxWallet,
    #[msg("Exceeds maximum transaction amount")]
    ExceedsMaxTx,
    #[msg("Cooldown period not met")]
    CooldownNotMet,
}
```

### Gotchas

- Hook program must be registered at mint creation — can't add later
- Extra accounts must be included in every transfer transaction
- Higher compute costs
- Breaks composability if extra accounts aren't passed correctly
- DEX integration requires the DEX to pass hook accounts (most major DEXes do in 2026)

## Metadata Pointer + Token Metadata

On-chain metadata without Metaplex. Built into Token-2022.

### Setup

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --enable-metadata \
  --name "My Token" \
  --symbol "MTK" \
  --uri "https://arweave.net/metadata.json"
```

### TypeScript

```typescript
import {
  createInitializeMetadataPointerInstruction,
  createInitializeMintInstruction,
  createUpdateFieldInstruction,
  ExtensionType,
  getMintLen,
  getTokenMetadata,
  TYPE_SIZE,
  LENGTH_SIZE,
} from "@solana/spl-token";
import { pack, TokenMetadata } from "@solana/spl-token-metadata";

const metadata: TokenMetadata = {
  mint: mint.publicKey,
  name: "My Token",
  symbol: "MTK",
  uri: "https://arweave.net/metadata.json",
  additionalMetadata: [["description", "A great token"]],
};

const metadataLen = TYPE_SIZE + LENGTH_SIZE + pack(metadata).length;
const extensions = [ExtensionType.MetadataPointer];
const mintLen = getMintLen(extensions);

// Create account, init metadata pointer, init mint, then write metadata
const initMetadataPointer = createInitializeMetadataPointerInstruction(
  mint.publicKey,
  authority,
  mint.publicKey, // Metadata lives on the mint itself
  TOKEN_2022_PROGRAM_ID
);
```

### When to Use

- Any new Token-2022 token (it's free, no reason not to)
- Replaces Metaplex for basic metadata (name, symbol, URI)
- Can store additional key-value pairs on-chain

### Gotchas

- Metadata is immutable unless the update authority is set
- Additional metadata fields have size limits
- Not all explorers/indexers read Token-2022 metadata yet (improving)
- Can still use Metaplex on top if needed

## Interest-Bearing Tokens

Display accrued interest in UIs without changing actual balance.

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --interest-rate 500   # 5.00% APR
```

### How It Works

- The actual token balance doesn't change
- A UI can calculate and display the accrued interest
- Rate can be updated by the rate authority
- Compound interest is NOT automatic — it's display-only

### When to Use

- Bond tokens / fixed-income tokens
- Yield-bearing stablecoins
- Display interest accrual in wallet UIs

## Non-Transferable Tokens (Soulbound)

Tokens that cannot be transferred. Can only be burned (if burn authority is set).

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --enable-non-transferable
```

### When to Use

- Achievement/credential NFTs
- Identity tokens
- Governance participation badges
- Soulbound tokens (SBTs)
- Proof-of-attendance

## Mint Close Authority

Allows closing the mint account and recovering rent.

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --enable-close
```

### Gotchas

- Mint can only be closed if supply is 0 (all tokens burned)
- Once closed, no more tokens can ever be minted
- Recovers ~0.002 SOL in rent

## Required Memo

Transfers fail unless a memo instruction precedes them.

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --enable-required-memo
```

### When to Use

- Compliance (every transfer needs a reference/note)
- Institutional tokens where audit trail is required
- Payment systems needing invoice references

### Gotchas

- DEXes must include memo instructions (most do)
- Adds complexity to every transfer
- Users must remember to include memo

## CPI Guard

Prevents other programs from transferring/burning tokens on your behalf via CPI.

```bash
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --enable-cpi-guard
```

### How It Works

- CPI (Cross-Program Invocation) calls that try to transfer your tokens are blocked
- Only direct instructions signed by you work
- Prevents malicious programs from draining your wallet

### When to Use

- Default security measure for user-facing tokens
- Prevents drainer programs from moving tokens via CPI

### Gotchas

- Breaks some DeFi composability (programs can't move your tokens automatically)
- Must be explicitly set per token account (not per mint)
- Users can enable/disable on their own accounts

## Migration from SPL Token to Token-2022

There is **no in-place migration**. Token-2022 is a different program.

### Migration Strategy

```
1. Create new Token-2022 mint with desired extensions
2. Snapshot existing token holders (on-chain query)
3. Announce migration with timeline
4. Users burn old tokens or deposit to migration contract
5. Migration contract mints new Token-2022 tokens 1:1
6. After grace period, revoke old mint authority
```

### Migration Contract Snippet

```rust
pub fn migrate_tokens(ctx: Context<MigrateTokens>, amount: u64) -> Result<()> {
    // Burn old SPL tokens
    let cpi_accounts = Burn {
        mint: ctx.accounts.old_mint.to_account_info(),
        from: ctx.accounts.user_old_token.to_account_info(),
        authority: ctx.accounts.user.to_account_info(),
    };
    token::burn(
        CpiContext::new(ctx.accounts.old_token_program.to_account_info(), cpi_accounts),
        amount,
    )?;

    // Mint new Token-2022 tokens
    let cpi_accounts = MintTo {
        mint: ctx.accounts.new_mint.to_account_info(),
        to: ctx.accounts.user_new_token.to_account_info(),
        authority: ctx.accounts.mint_authority.to_account_info(),
    };
    token_2022::mint_to(
        CpiContext::new(ctx.accounts.new_token_program.to_account_info(), cpi_accounts),
        amount,
    )?;

    Ok(())
}
```

### Migration Best Practices

- Give 90+ days for migration
- Use a one-way bridge (burn old → mint new)
- Keep old token visible but non-functional after deadline
- Document migration in all official channels
- Consider automated migration via Jupiter swap pairs

## CLI Commands Quick Reference

```bash
# Create Token-2022 mint with multiple extensions
spl-token create-token \
  --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb \
  --decimals 6 \
  --enable-metadata \
  --name "My Token" \
  --symbol "MTK" \
  --uri "https://arweave.net/meta.json" \
  --transfer-fee 50 5000

# Initialize mint
spl-token create-account <MINT> --program-id TokenzQdBNbLqP5VEhdkAS6EPFLC1PHnBqCXEpPxuEb
spl-token mint <MINT> 1000000000

# Update transfer fee config
spl-token authorize <MINT> transfer-fee-config <NEW_AUTHORITY>

# Withdraw withheld transfer fees
spl-token withdraw-withheld-tokens <DESTINATION> <MINT>

# Update metadata
spl-token update-metadata <MINT> name "New Name"
spl-token update-metadata <MINT> uri "https://new-uri.com/meta.json"

# Freeze/thaw accounts
spl-token freeze <ACCOUNT>
spl-token thaw <ACCOUNT>

# Check account info (includes extensions)
spl-token account-info <ACCOUNT>

# List all mints with extensions
spl-token display <MINT>
```

## Extension Selection Decision Matrix

| Need                        | Extension(s)                        | Trade-off          |
|-----------------------------|-------------------------------------|--------------------|
| Revenue on transfers        | Transfer Fee                        | Reduces received   |
| Privacy                     | Confidential Transfer               | High compute cost  |
| KYC/allowlist              | Default Account State + Freeze Auth | Centralized        |
| Anti-whale                  | Transfer Hook                       | Composability risk |
| Metadata on-chain           | Metadata Pointer                    | None (free)        |
| Compliance audit trail      | Required Memo                       | UX friction        |
| Recoverable tokens          | Permanent Delegate                  | Centralization     |
| Soulbound/credentials       | Non-Transferable                    | No secondary market|
| Display yield               | Interest Bearing                    | Display-only       |
| Security (anti-drainer)     | CPI Guard                           | DeFi composability |
| Recover rent                | Mint Close Authority                | Requires 0 supply  |

**Default recommendation for 2026:** Metadata Pointer + Mint Close Authority. Add others only when specific requirements exist.
